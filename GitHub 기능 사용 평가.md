# 마일스톤 (Milestones)
목표: 프로젝트의 주요 단계와 기한을 설정하여 진행 상황을 추적한다.
목표 설정,"성공: 1차 디자인 완료 마일스톤을 설정하고 관련 이슈들을 묶었더니, 팀원 모두가 현재 시점에 달성해야 할 목표를 명확히 인지하고 작업 속도를 조절할 수 있었다.
실패: 마일스톤의 범위를 너무 넓거나(예: 전체 프로젝트) 너무 좁게(예: 단순 버그 수정) 설정하여, 기한 관리가 비효율적이었고 마일스톤 기능 자체를 사용하지 않게 되었다.

진행 추적,"성공: 마일스톤 진행률(%)이 자동으로 표시되어, 팀장/교수님이 프로젝트의 지연 여부를 쉽게 파악할 수 있었고, 문제가 있는 팀원에게 즉시 도움을 줄 수 있었다.
실패: 이슈를 마일스톤에 지정하지 않는 팀원들이 발생하여, 실제 진행률과 마일스톤의 표시 진행률이 달라져 신뢰성을 잃었다.

# 레이블 (Labels)
목표: 이슈의 성격, 중요도, 담당자 등을 분류하여 관리 효율을 높인다.
작업 분류,"성공: bug, feature, documentation 레이블을 사용해 이슈 목록을 필터링했더니, 현재 시급하게 처리해야 할 버그와 개발해야 할 새로운 기능을 즉각 구분하여 우선순위를 설정할 수 있었다.
실패: 레이블을 너무 많이(10개 이상) 만들거나, 명칭을 통일하지 않아 (예: design-ui와 UI 디자인) 오히려 혼란만 가중되었고, 팀원들이 레이블을 붙이는 것을 귀찮아하게 되었다.

담당/우선순위,"성공: P1-Critical, P4-Low 등의 우선순위 레이블을 사용해 중요도에 따라 작업 시간을 배분하는 데 큰 도움이 되었다.
실패: 레이블을 만든 후, 팀원들에게 레이블 사용 규칙을 명확히 공지하지 않아, 대부분의 이슈에 레이블이 아예 붙지 않거나 잘못된 레이블이 붙어 기능이 무용지물이 되었다.

# 풀 리퀘스트 (Pull Requests, PR)
목표: 코드/콘텐츠 변경 사항에 대한 검토(리뷰) 과정을 통해 품질을 관리하고 기록을 남긴다.
평가 항목,성공 사례 (장점),실패/어려움 사례 (단점)
코드/파일 리뷰,"성공: PR 리뷰 기능을 통해 팀원들이 서로의 작업 내용을 검토하면서 디자인 오류나 오타, 누락된 부분을 발견하여 메인 파일에 반영하기 전에 수정할 수 있었다. (품질 관리 기여)
실패: PR에 대한 리뷰 없이 'Merge' 버튼만 누르는 팀원이 많아져, PR 과정이 형식적인 절차로 전락했고, 잘못된 내용이 메인 브랜치에 그대로 반영되었다.

히스토리 관리,"성공: PR을 병합(Merge)할 때, 관련 이슈 번호와 기능 이름을 제목에 명시하여 main 브랜치의 커밋 히스토리가 기능 단위로 깔끔하게 정리되었다. (👉 위 검색 결과의 장점과 유사)
실패: PR 하나에 여러 개의 독립적인 기능을 한꺼번에 담아 요청하여, 리뷰어가 검토하기 어렵고 문제가 발생했을 때 어떤 기능 때문인지 추적하기 어려웠다."
